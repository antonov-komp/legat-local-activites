## Описание
Реализовать кастомное activity для бизнес-процессов Битрикс24, которое получает сведения о заявлениях о возбуждении приказного производства и делах, зарегистрированных в экономическом суде в рамках приказного производства, через метод Legat `GET /api2/by/court`.

Activity должно принимать УНП компании и возвращать в дополнительные выходные поля структурированные данные по делам, включая роль компании (взыскатель/должник), реквизиты дела, решение суда, сумму, стороны, суд и признак отказа в принятии.

## Контекст
В проекте уже есть рабочее activity для `GET /api2/by/data` и подготовлена задача для `GET /api2/by/contacts`. Нужно сделать аналогичный рабочий шаблон для судебных данных (`court`) с учетом:
- пагинации (`page`);
- типа стороны (`type`: 1 — взыскатель, 2 — должник);
- возможного пустого результата (`total = 0`, `claimant = null`).

Ключ API должен оставаться скрытым и загружаться из серверной конфигурации:
- `local/legat_api_key.php` (приоритет),
- fallback: `Option::get('main', 'legat_by_data_api_key')`.

## API-методы (Legat)
- `GET /api2/by/court`
- Полный URL: `https://api.legat.by/api2/by/court`

Параметры запроса:
- `key` (string, required) — ключ доступа
- `unp` (string, required) — УНП
- `type` (integer, optional, enum: `1`, `2`) — тип стороны:
  - `1` — компания в роли взыскателя
  - `2` — компания в роли должника
- `page` (integer, optional) — страница

Пример запроса:
`https://api.legat.by/api2/by/court?unp=692172505&key=***`

Пример ответа:
```json
{
  "error": null,
  "total": 0,
  "claimant": null
}
```

## Технические требования
- Платформа: Bitrix24 (custom activity).
- Реализация: PHP, без фреймворков.
- Ключ API не выводить в UI и не хранить в репозитории.
- Поддержать сценарий пустых данных (`total = 0`, список дел отсутствует).
- Возвращать данные в стабильной структуре, пригодной для дальнейшего использования в БП.
- Обязательное поле `REQUEST_STATUS` в любом исходе выполнения.
- Обязательно вернуть `TOTAL` (общее число записей по ответу API).

## Предлагаемая структура выходных полей
- `REQUEST_STATUS` — статус (`Успешно` / текст ошибки)
- `TOTAL` — общее количество записей
- `COURT_ITEMS` — нормализованный список дел (массив или JSON-строка)
- `COURT_ROLE` — примененный тип стороны (`1`/`2`, если задан)
- `COURT_PAGE` — обработанная страница
- `COURT_RAW_JSON` — опционально сырой JSON-ответ для отладки
