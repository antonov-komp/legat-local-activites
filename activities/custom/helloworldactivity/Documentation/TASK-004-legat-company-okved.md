## Описание
Реализовать кастомное activity для бизнес-процессов Битрикс24, которое получает сведения об актуальных и исторических видах деятельности компании из API Legat по УНП через метод `GET /api2/by/okved` и возвращает нормализованный список записей ОКВЕД в дополнительные выходные поля activity.

## Контекст
В проекте уже реализованы скелеты/планы для Legat: `data`, `contacts`, `court`. Нужно добавить метод `okved()` по тому же принципу без изменения существующих activity.

Ключ API не отображается в UI и берётся из:
- `local/legat_api_key.php` (приоритет),
- fallback: `Option::get('main', 'legat_by_data_api_key')`.

## Ступенчатые подзадачи
1. Создать activity `legatcompanyokved` по образцу рабочего `legatcompanyinfo`.
2. Настроить `.description.php`: вход — УНП; выходы — статус, `total`, список ОКВЕД (и при необходимости сырой JSON).
3. Реализовать GET `https://api.legat.by/api2/by/okved?unp=...&key=...` (ключ только на сервере).
4. Распарсить JSON: поля `error`, `total`, `okved` (массив).
5. Для каждого элемента `okved[]` смаппить поля:
   - `name` — наименование ОКВЕД;
   - `code` — код ОКВЕД;
   - `date_begin` — дата внесения сведений в ЕГР об ОКВЕД;
   - `date_end` — дата внесения в ЕГР об изменении ОКВЕД (для исторических; в API может приходить `"0000-00-00"` — трактовать как «нет даты окончания» или пустое значение в выходе);
   - `active` — признак: `1` действующий ОКВЕД, `0` исторический.
6. Обработать пустой результат: `total = 0` и/или пустой массив `okved`.
7. Реализовать обработку ошибок: отсутствие ключа, пустой УНП, HTTP, невалидный JSON, непустое `error` в ответе.
8. Добавить логирование без утечки `key`.
9. Протестировать в конструкторе БП.

## API-методы (Legat)
- `GET /api2/by/okved`
- URL: `https://api.legat.by/api2/by/okved`

Параметры:
- `key` (string, required) — ключ доступа
- `unp` (string, required) — УНП

Пример запроса:  
`https://api.legat.by/api2/by/okved?unp=692172505&key=***`

Пример ответа:
```json
{
  "error": null,
  "total": 2,
  "okved": [
    {

      "code": 46900,
      "date_begin": "2020-11-13",
      "date_end": "0000-00-00",
      "active": 1
    },
    {
      "name": "Прочая розничная торговля вне магазинов, палаток, киосков и рынков",
      "code": 47990,
      "date_begin": "2020-11-13",
      "date_end": "0000-00-00",
      "active": 0
    }
  ]

```

## Технические требования
- PHP, кастомное activity Битрикс24; ключ не в репозитории и не в UI.
- Выходные данные — структурированные (массив в `preparedProperties` или согласованная JSON-строка для шагов БП — зафиксировать в README).
- Поле `REQUEST_STATUS` обязательно при любом исходе.
- Отдавать `TOTAL` из ответа API.
- Нормализовать даты: для `0000-00-00` не передавать ложные даты в последующие шаги без явной документации (рекомендуется пустая строка или `null` в нормализованном выводе).

## Предлагаемая структура выходных полей
- `REQUEST_STATUS` — успех или текст ошибки
- `TOTAL` — число из поля `total`
- `OKVED_ITEMS` — список записей с полями: `name`, `code`, `date_begin`, `date_end`, `active`
- `OKVED_ACTIVE_ONLY` — опционально: только записи с `active === 1`, если удобно для БП
- `OKVED_RAW_JSON` — опционально: сырой ответ для отладки
